# 用户维度 API 监控仪表盘 - 设计讨论方案

> 创建日期: 2026-03-28
> 目的: 便于快速发现系统问题和异常用户
> 涉及部门: 产品经理、UI前端、开发部、数据分析、运维、安全

---

## 1. 业务背景与目标

### 1.1 核心诉求
- **系统层面**: 快速识别系统整体健康度、API 质量问题
- **用户层面**: 快速发现异常用户（如恶意刷接口、账户被盗用、配置错误等）

### 1.2 关键指标定义

| 指标 | 计算方式 | 业务意义 |
|------|----------|----------|
| API请求数 | COUNT(api_logs.id) | 用户活跃度、使用量 |
| Token消耗 | SUM(input_tokens + output_tokens) | 资源消耗、成本 |
| 请求失败率 | failed_calls / total_calls × 100% | API质量、用户使用正确性 |
| 请求失败数 | COUNT(status != 'success') | 绝对影响面 |

### 1.3 时间维度
- 实时: 最近 5 分钟
- 短期: 今日/本周
- 长期: 近 30 天趋势

## 2. 图表类型讨论

### 2.1 候选图表及适用场景

| 图表类型 | 优点 | 缺点 | 推荐场景 |
|----------|------|------|----------|
| **柱状图** | 对比清晰、易理解 | 不适合时间序列 | Top N 用户排名 |
| **折线图** | 趋势展示直观 | 多线时易混乱 | 单用户/全系统时间趋势 |
| **饼图** | 比例直观 | 分类过多时混乱 | 失败类型占比 |
| **散点图** | 发现异常值 | 需要解释 | 用户分布、异常用户筛选 |
| **热力图** | 模式识别强 | 颜色感知差异 | 时间+用户交叉分析 |
| **表格+迷你图** | 精确数据+趋势 | 信息密度高 | 详细数据分析 |

## 3. 功能模块设计

### 3.1 模块划分

```
用户 API 监控
├── 全局统计概览
│   ├── 今日/本周/本月 核心指标
│   └── 系统健康度评分
│
├── 排行榜分析
│   ├── 请求量 Top N
│   ├── Token 消耗 Top N
│   ├── 失败率 Top N (降序)
│   └── 新增用户 Top N
│
├── 用户明细表
│   ├── 支持排序/筛选
│   ├── 点击展开详情
│   └── 批量操作
│
├── 异常检测
│   ├── 失败率异常 (阈值可配置)
│   ├── 请求量突增 (环比/同比)
│   ├── Token 消耗异常
│   └── 沉默用户 (长期无活动)
│
└── 用户详情页
    ├── 每日趋势折线图
    ├── 请求类型分布
    ├── 失败原因分析
    └── 操作日志
```

### 3.2 筛选条件

| 筛选维度 | 选项 | 说明 |
|----------|------|------|
| 时间范围 | 今日/本周/本月/自定义 | 必选 |
| 用户等级 | 全部/免费/VIP/企业 | 辅助分析 |
| 状态 | 全部/正常/异常/已封禁 | 快速定位 |
| 指标排序 | 请求量/Token/失败率 | 默认失败率降序 |

### 3.3 告警规则 (可配置)

| 规则 | 默认阈值 | 动作 |
|------|----------|------|
| 失败率过高 | > 30% | 警告 |
| 失败率严重 | > 50% | 严重警告 |
| 请求量突增 | 环比 > 300% | 警告 |
| 请求量异常 | 超过日限额 80% | 警告 |
| Token 消耗异常 | 超过配额 90% | 警告 |

### 3.4 扩展监控维度 (优化建议)

#### 3.4.1 安全风控维度

| 指标 | 业务意义 | 异常阈值建议 |
|------|----------|--------------|
| IP聚集度 | 同一IP多用户访问 | > 5用户/IP |
| 请求来源分布 | 是否有爬虫/自动化脚本 | 请求间隔 < 100ms |
| 异常时间访问 | 非工作时间大量请求 | 凌晨2-6点 > 1000请求 |
| API调用模式 | 是否符合正常用户行为 | 短时间请求相同内容 |
| 并发异常 | 同时大量请求 | 并发数 > 100 |
| 地区分布 | 用户使用地区统计 | 异常地区告警 |

#### 3.4.2 成本分析维度

| 指标 | 计算方式 | 业务意义 |
|------|----------|----------|
| 人均成本 | Token消耗 × 单价 / 用户数 | 定价策略参考 |
| 用户LTV | 用户生命周期价值 | 留存分析 |
| ROI分析 | 用户贡献 vs 资源消耗 | 用户价值评估 |
| 模型使用分布 | 各模型消耗占比 | 成本结构优化 |
| 每日成本趋势 | 按日统计总成本 | 成本监控 |

#### 3.4.3 用户行为分析

| 指标 | 描述 | 用途 |
|------|------|------|
| 活跃时间段 | 用户主要使用时段 | 容量规划 |
| 平均响应时间 | API平均延迟 | 性能监控 |
| 模型偏好 | 用户使用模型分布 | 模型资源配置 |
| 请求复杂度 | 平均输入/输出Token | 资源预估 |
| 重复请求率 | 相同请求比例 | 缓存优化参考 |

#### 3.4.4 系统健康度评分

```
综合评分 = (成功率 × 40%) + (响应速度 × 30%) + (用户满意度 × 30%)

评分等级:
- 90-100: 🟢 优秀
- 70-89:  🟡 良好
- 50-69:  🟠 一般
- < 50:   🔴 需关注
```

#### 3.4.5 报表与导出功能

| 功能 | 格式 | 内容 |
|------|------|------|
| 日报 | PDF/Excel | 每日核心指标汇总 |
| 周报 | PDF/Excel | 周趋势 + Top异常用户 |
| 月报 | PDF/Excel | 月度分析 + 成本报表 |
| 自定义导出 | Excel/CSV | 支持自定义时间/筛选条件 |
| 定时发送 | 邮件 | 可配置定时发送给管理员 |

## 4. 数据统计方案

### 4.1 数据来源

```sql
-- 基于 api_logs 表统计
SELECT 
    user_id,
    COUNT(*) as total_calls,
    SUM(input_tokens + output_tokens) as total_tokens,
    SUM(CASE WHEN status != 'success' THEN 1 ELSE 0 END) as failed_calls,
    ROUND(SUM(CASE WHEN status != 'success' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as failure_rate
FROM api_logs
WHERE created_at >= :start_time AND created_at < :end_time
GROUP BY user_id
```

### 4.2 预聚合 vs 实时统计

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 实时统计 | 数据准确 | 查询慢 | 数据量 < 100万/天 |
| 预聚合 (小时表) | 查询快 | 有延迟 | 数据量 > 100万/天 |
| 预聚合 (天表) | 最快 | 粒度粗 | 历史分析 |

**建议**: 初期使用实时统计，数据量增长后迁移到预聚合

### 4.3 性能考虑

- 用户量 < 1000: 实时查询可接受
- 用户量 1000-10000: 缓存 + 增量更新
- 用户量 > 10000: 预聚合 + 分页

## 10. 技术风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| 数据量大导致查询慢 | 用户体验差 | 预聚合 + 缓存 + 分页 |
| 实时性要求高 | 数据库压力大 | Redis缓存 + 异步处理 |
| 异常检测误报 | 打扰正常用户 | 多维度验证 + 白名单 |
| 历史数据膨胀 | 存储成本高 | 分区表 + 冷热分离 |
| 图表渲染卡顿 | 大量数据点 | 数据采样 + 分页加载 |
